# Smart API Gateway — 3分钟面试讲解稿

---

大家好，我带来的项目是 **Smart API Gateway**，一个集限流、监控、异常检测于一体的智能 API 网关平台。

**背景与目标**

在现代微服务架构中，网关是流量的第一道关口。我的目标是从零实现一个生产可用的网关，而不是简单调用现成的限流库。我希望通过这个项目，深入理解限流算法的本质，同时把异常检测、实时监控这些运维能力也集成进去。

**技术栈**

后端采用 Node.js + Express 5，数据库用 better-sqlite3 同步驱动并开启 WAL 模式保证并发写入性能。前端是原生 HTML/CSS/JS 搭配 Chart.js 做可视化，实时推送用 WebSocket 实现。整个项目是 ES 模块组织，没有用 TypeScript，保持轻量。

**核心功能**

最核心的是三种限流算法的自实现。固定窗口用计数器加窗口起始时间，到期清零；滑动窗口维护每个 IP 的时间戳数组，每次请求清理过期元素再判断；令牌桶按恒定速率填充令牌，支持一定程度的突发流量。三种算法可以通过管理界面实时切换，无需重启服务。

第二个亮点是异常检测模块。我用了统计学里的移动平均加标准差方法——每 5 秒采样最近 5 秒的平均 QPS，学习期 30 秒建立基线均值和标准差，之后任何超过 2 倍标准差的偏离都会被判定为异常。超过 2σ 是 warning，超过 3σ 是 critical，异常事件通过 WebSocket 即时推送到前端弹出红色告警横幅。这个设计借鉴了工业界常用的 3-sigma 规则，简洁但有效。

第三个是完整的工程化实践。从数据库的预编译语句、配置热更新缓存、IP 黑白名单的 30 秒 TTL 缓存，到异步日志写入避免阻塞请求、WebSocket 断线自动降级 HTTP 轮询，再到 autocannon 压测工具的三算法对比和异常检测验证脚本——这些细节让项目从一个 demo 变成了真正可以演示的完整系统。

**压测数据**

我用 autocannon 跑了三组测试：突发流量场景下令牌桶表现最优，429 分布平缓；边界攻击测试中滑动窗口比固定窗口精确得多，不会出现窗口边界的双倍放行问题。具体数据在压测报告里有详细记录。

**我的收获**

这个项目让我真正理解了"限流不是简单计数"这件事。比如固定窗口的边界突刺问题、令牌桶的预热问题，只有亲手实现过才会明白为什么 Redis 和 Sentinel 选择滑动窗口作为默认算法。同时也锻炼了我的全栈能力——从数据库设计到前端仪表盘，从统计方法到压测分析，是一个串起来的完整工程。
