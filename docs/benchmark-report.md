## 算法特性分析

### 固定窗口 (Fixed Window)

```
原理：在固定时间窗口内计数，窗口结束计数器归零

优点：
  - 实现最简单，内存占用低（每个 IP 只存 count + windowStart）
  - 窗口边界清晰，便于理解和调试

缺点：
  - 窗口边界的"双倍流量"问题：窗口末尾和下一个窗口开头的请求都被放行
  - 流量分布不均，可能出现周期性突发

适用场景：
  - 对精度要求不高的粗粒度限流
  - 不需要处理突发流量的简单 API
```

### 滑动窗口 (Sliding Window)

```
原理：维护请求时间戳数组，清理过期记录，判断当前窗口内请求数

优点：
  - 平滑限流，无"双倍流量"问题
  - 精度高，任意时间点的 QPS 都不会超过限额
  - 配置驱动，支持热更新

缺点：
  - 内存占用随请求量增长（需存储时间戳数组）
  - 滑动窗口清理操作在极端高并发下有性能开销

适用场景：
  - 需要精确平滑限流的 API 网关
  - 严格 SLA 要求的服务接口
  - 主流推荐方案，适合大多数场景
```

### 令牌桶 (Token Bucket)

```
原理：桶以固定速率填充令牌，请求消费令牌，桶空则拒绝

优点：
  - 天然支持突发流量：空闲时攒满令牌，突发时可一次性消费
  - 长期平均速率可控（由 fillRate 决定），短期允许突发（由 capacity 决定）
  - 内存占用最低（每个 IP 只存 tokens + lastRefill）

缺点：
  - 配置两个参数（capacity 和 fillRate），需要理解二者关系
  - 突发结束后需要等待令牌补充

适用场景：
  - 允许合理突发的 API（如秒杀、热点事件）
  - 需要兼顾平均限流和突发容量的场景
  - 推荐用于有突发容忍需求的业务接口
```

## 结论

| 场景 | 推荐算法 | 理由 |
|------|----------|------|
| 简单 API 限流 | 滑动窗口 | 平滑精确，配置直观 |
| 允许突发 | 令牌桶 | capacity 控制突发上限，fillRate 控制长期速率 |
| 低内存环境 | 固定窗口 | 每个 IP 仅 2 个字段，无时间戳数组 |
| 严格 SLA | 滑动窗口 | 任意时刻不超过限额，无窗口边界问题 |
| 秒杀/热点 | 令牌桶 | 允许一时突发，过后平滑恢复 |

### 综合建议

- **默认选择滑动窗口**：精度高、配置简单、行为可预测
- **高突发场景选择令牌桶**：充分利用系统资源，用户体验更好
- **固定窗口仅用于简单场景**：或作为降级方案（Redis 等外部存储不可用时）
